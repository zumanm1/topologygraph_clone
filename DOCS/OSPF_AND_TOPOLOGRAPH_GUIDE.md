# OSPF LSDB, Topolograph, and Location-Based Clustering

## 1. What is Topolograph?

[Topolograph](https://github.com/Vadims06/topolograph) is a web-based tool that:

- **Builds OSPF/IS-IS topology** from Link-State Database (LSDB) output from a single router.
- **Visualizes** the network as a graph: routers = nodes, OSPF adjacencies = edges with costs.
- **Supports analysis**: shortest path, backup paths, link/node failure simulation, cost planning, asymmetric path detection, heatmaps.

Because OSPF floods LSAs to all routers in an area, one device has the full area LSDB. You run vendor-specific commands (e.g. Cisco: `show ip ospf database router` / `network` / `external`), save the output to a file, and upload it to Topolograph (web or API). Topolograph parses the file with TextFSM-style templates and builds the graph.

**Important:** For Cisco OSPF, Topolograph expects **both** LSA Type 1 (Router) and LSA Type 2 (Network) in the same file. Your current `ospf-database.txt` contains only Router LSAs. For full compatibility, append the output of:

```bash
show ip ospf database network
```

to `ospf-database.txt` and upload that combined file to [topolograph.com](https://topolograph.com) or your Docker instance.

The LSDB contains only **Router IDs** (IP addresses, e.g. `9.9.9.1`, `18.18.18.6`). It does **not** contain hostnames. So “location” cannot be read from the LSDB alone; it must come from a separate mapping.

## 3. Clustering Nodes by Location (First 3 Letters = Country)

You specified that each router’s hostname starts with a **3-letter country/location code** (e.g. `zaf`, `ngr`, `ken`). To cluster nodes by location we need **Router ID → hostname**. That is provided by a **mapping file** (CSV).

### 3.1 Mapping file format

CSV with two columns (order or header name can vary):

- `router_id` (or `rid`): OSPF Router ID (e.g. `18.18.18.6`).
- `hostname` (or `name`): Router hostname (e.g. `zaf-jnb-r2`).

The script takes the **first 3 characters** of the hostname (case-insensitive) as the **location code** (e.g. `zaf` → ZAF, `ngr` → NGR).

### 3.2 How to get the mapping

- Export from your NMS/CMDB or inventory (e.g. Ansible, NetBox).
- Or fill the template generated by the script:  
  `python3 parse_ospf_lsdb.py ospf-database.txt`  
  This creates `router_id_to_hostname.csv` with all Router IDs and empty hostnames; you fill the hostname column.

### 3.3 Running the parser with mapping

```bash
# Cluster by location and print summary
python3 parse_ospf_lsdb.py ospf-database.txt -m hostname_mapping_example.csv

# Also write Topolograph YAML with location tags
python3 parse_ospf_lsdb.py ospf-database.txt -m hostname_mapping_example.csv -o topology_with_locations.yaml
```

With the example mapping you get **9 locations** (EGY, GHA, KEN, MAR, NGR, TZA, UGA, ZAF, etc.) and a YAML where each node has a `location` tag. Topolograph can filter nodes by this tag via its API (e.g. “all nodes in ZAF”).

## 4. Using the Generated YAML with Topolograph

From **v2.32**, Topolograph accepts **YAML topologies**: you define `nodes` and `edges`; nodes can have optional **tags** (e.g. `location: ZAF`).

- **Upload:** In the Topolograph UI, use “Upload” and select the YAML file (e.g. `topology_with_locations.yaml`).
- **API:** Use the diagram/upload endpoint with the YAML body.

In the YAML we produce:

- **nodes:** `name` = Router ID (IP format), `label` = hostname (for display), `location` = 3-letter code (for filtering).
- **edges:** `src`, `dst`, `cost` from the parsed Router LSAs (point-to-point links only).

So you can:

- Visualize the same topology as in the LSDB.
- Filter or group nodes by `location` (e.g. “show only ZAF”) in the UI or via the API (e.g. `GET /api/diagram/{graph_time}/nodes?location=ZAF`).

## 7. Theory in Brief: Why This Works

- **OSPF** distributes **Link-State Advertisements (LSAs)** so every router in an area builds the same **LSDB** and runs **SPF** to get a shortest-path tree. Router LSAs (Type 1) describe a router’s links and costs; Network LSAs (Type 2) describe multi-access segments and the DR. So one router’s LSDB dump is enough to reconstruct the area topology.
- **Topolograph** parses that dump, builds a graph, and runs the same kind of shortest-path and failure-simulation logic you would on a real network.
- **Hostnames** are not carried in OSPF LSAs; they are a local/operational detail. So **location = f(hostname)** requires a separate Router ID → hostname mapping. Using the first 3 characters as a country/location code is a simple, consistent convention for clustering and filtering in Topolograph.

You now have a clear path from your `ospf-database.txt` to a topology that Topolograph can visualize and that you can cluster and filter by location using the 3-letter hostname prefix.
